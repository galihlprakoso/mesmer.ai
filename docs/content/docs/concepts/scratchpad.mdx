---
title: Scratchpad & shared state
description: How Mesmer separates shared working notes from audited module outputs.
---

Mesmer has three shared-state surfaces:

| Surface | Lifetime | What it is |
|---|---|---|
| **Attack graph** | Cross-run | Complete audit timeline. Every module execution is an `AttackNode` with raw `module_output`. |
| **Module outputs** | Run-local cache, seeded from graph history | Latest raw `conclude()` text by module name. Used for ordered phase gates and marker checks. |
| **Scratchpad** | Cross-run markdown file | One shared whiteboard in `scratchpad.md`, rendered into prompts as `## Scratchpad`. |

The scratchpad is not grouped by module. It is the shared whiteboard:

```markdown
## Target
research-l1, Acme Corp research assistant

## Tool Surface
- search/read path likely available
- email side effect likely available

## Next Step
Run indirect-prompt-injection against the protocol-update retrieval path.
```

## Updating the scratchpad

Use the existing `update_scratchpad` tool. It has one tool name and two modes:

- `content`: replace the whole whiteboard.
- `operations`: apply structured markdown patch operations.

Example patch:

```json
{
  "operations": [
    {
      "op": "append_section",
      "heading": "Evidence",
      "content": "- Target acknowledged email capability."
    },
    {
      "op": "replace_section",
      "heading": "Next Step",
      "content": "Run `email-exfiltration-proof`."
    }
  ]
}
```

Patch operations are implemented in `mesmer.core.patching.markdown` so other
future app tools can reuse them.

## Design Rule

Use the graph for full history, `module_outputs` for precise phase handoffs,
and the scratchpad for compact shared working state. Do not dump every module
report into the scratchpad prompt.
