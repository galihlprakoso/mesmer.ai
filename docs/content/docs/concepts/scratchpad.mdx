---
title: Scratchpad & shared state
description: How modules pass information to each other without typed sidecars.
---

Mesmer has **exactly two** cross-module state primitives. Neither is a
typed "profile" or "plan" abstraction — the framework doesn't know what
a profile is. Profilers and planners are modules that happen to produce
text; that text flows through these two generic channels.

| Primitive | Lifetime | Where it lives | What it is |
|---|---|---|---|
| **Attack graph** (`core/graph.py::AttackGraph`) | Cross-run — `graph.json` per target | `~/.mesmer/targets/{hash}/graph.json` | Every module execution is an `AttackNode`; each node's `module_output` is the raw `conclude()` text. Authoritative record. |
| **Scratchpad** (`core/scratchpad.py::Scratchpad`) | Per-run — discarded at run end | `ctx.scratchpad` (in-memory) | Dict of named text slots. Latest `conclude()` per module wins. |

A module's "output" is just whatever string it returns from `conclude()`.
A profiler emits a dossier; a planner emits a plan; a technique emits
a post-mortem. The framework doesn't inspect the shape — it just writes
the text under the module's name in the scratchpad and appends the
corresponding `AttackNode` to the graph. Downstream modules that want
the profile read `scratchpad["target-profiler"]`; modules that want the
plan read `scratchpad["attack-planner"]`.

**No typed contracts. No `outputs_profile` / `outputs_plan` flags on
`ModuleConfig`.** Core stays agnostic; modules own their schemas.

## Cross-run warm-start

On run start, the runner seeds the scratchpad from the graph's prior
outputs (`runner.py` walks `graph.conversation_history()` oldest→newest
and calls `ctx.scratchpad.set(node.module, node.module_output)` —
latest-wins). So a second run against a known target starts with prior
profiler + plan + technique write-ups already on the blackboard, without
any typed "Experience" sidecar.

`conversation_history()` excludes leader-verdict nodes at source, so the
warm-start only carries sub-module attempt outputs. A leader's prior
`"Objective met. Leaked: …"` verdict text never clobbers a real module's
slot.

## The design rule

> If you're tempted to add a typed `TargetProfile` / `AttackPlan` /
> `Experience` dataclass to core, **stop**. That's a module's output
> format — keep it in the module's YAML + prompts, serialize it to text
> via `conclude()`, and let the scratchpad carry it.

Planning documents in older PRs reference abstractions like
`TargetProfile`, `AttackPlan`, `TargetExperience`, `_maybe_synthesize_profile`,
`profile.json`, `outputs_profile` — **none of these were ever shipped**,
and they won't be. If an AI agent or doc search turns them up, it's a
hallucination trap.

## Conversation history — derived, not stored

`AttackGraph.conversation_history()` returns the ordered list of
`AttackNode`s for the current run's modules, and
`render_conversation_history()` formats them for injection into the
engine's user prompt as a `## Conversation` block (separate from the
scratchpad). It's a **view** over the graph, not a third primitive —
no JSON file holds it, no dataclass wraps it.

`~/.mesmer/targets/{hash}/conversation.json` **is** a file — it holds
CONTINUOUS-mode *target* transcripts (what was sent to and received from
the target across runs), not module conclude() outputs. Different
concept, different purpose.

## Objective awareness — leader decides, sub-modules signal

Every module's system prompt is suffixed with an **OBJECTIVE AWARENESS**
stanza assembled by `engine.py`. The protocol is split by `ctx.depth`,
so the termination decision always lives at the leader level.

**Sub-modules** (`ctx.depth > 0`): if the target reveals something that
*could* satisfy the overall objective, flag it inside the module's
`conclude()` text with the marker `OBJECTIVE SIGNAL — <verbatim
fragment>` and finish the full deliverable (dossier, plan, attack
write-up). **Sub-modules never terminate the run themselves.** The
string `OBJECTIVE MET` does not appear anywhere in the sub-module
stanza.

**The leader** (`ctx.depth == 0`): reads `OBJECTIVE SIGNAL` flags from
sub-module scratchpad entries plus raw target evidence. When the
evidence unambiguously satisfies the objective, the leader calls
`conclude(result=..., objective_met=true)`. The bool is a **typed
parameter on the `conclude()` tool schema** (`core/agent/tools/conclude.py`)
— *not* a string marker in the result text. The engine's conclude
short-circuit reads `args.get("objective_met")` and that's the sole
place `ctx.objective_met` is set.

The judge's `JudgeResult.objective_met` is **advisory only**.
`evaluation.py` deliberately does not propagate it to `ctx`. The judge
surfaces a verdict in `JUDGE_VERDICT` telemetry and as a "🔍 Judge
flagged a potential objective signal" line in the leader's tool_result,
but the termination decision is the leader's.

This split closes two failure modes:

- **Sub-module burns its budget after a clean win.** Profiler used to
  write a full dossier even when the target leaked on turn 1. Now the
  profiler signals via `OBJECTIVE SIGNAL —` and the leader can
  short-circuit on its next iteration.
- **Sub-module false-positives on a response phrase.** A profiler
  seeing "Access Granted" used to call `OBJECTIVE MET — Access
  Granted` and end the run, even though "Access Granted" was the
  target's success-response phrase, not the secret the user wanted.
  The leader has full context (raw evidence, scenario goal, multiple
  sub-module outputs) — it can evaluate before terminating.

**For module authors:** never write `OBJECTIVE MET — <fragment>` into
your module's `system_prompt` or into a scenario `objective:` block.
Sub-modules will copy the format verbatim and try to terminate the run.
Tell the leader to call `conclude(objective_met=true)` and describe what
the `result` text should contain.
