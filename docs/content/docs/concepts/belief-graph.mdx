---
title: The belief graph
description: Mesmer's typed planner state — weakness hypotheses, evidence, and frontier experiments — that runs alongside the legacy attack graph.
---

The [legacy attack graph](./attack-graph) is an attempt log: every
module run becomes one node with a score and a status. That answers
"what did we try?" — not "what do we believe about this target?" The
**belief graph** is the typed alternative the planner runs on. Six
node kinds, eleven typed edges, and one mutation primitive
(`GraphDelta`). The agent's question shifts from "which module should
I run next?" to "which suspected weakness should I test next, with
which strategy, given what I already believe?"

The legacy attack graph is **not replaced**. Both run side-by-side
against every target — the belief graph is the planner substrate, the
attack graph remains the per-attempt audit trail. See
[the comparison](#side-by-side-with-the-legacy-attack-graph) at the
bottom of this page.

Every node carries `id` (kind-prefixed: `tg_…` `wh_…` `ev_…` `at_…`
`st_…` `fx_…`), `created_at`, `run_id`, plus its own typed payload:

| Kind | Holds | Why it's distinct |
|---|---|---|
| **Target** (`TargetNode`) | singleton; `traits: dict[str, str]` | Free-form recon dossier. Latest-write-wins per key. The schema is intentionally loose — the extractor and planner both read traits as opaque text. |
| **Weakness hypothesis** (`WeaknessHypothesis`) | `claim`, `family`, `confidence ∈ [0,1]`, `status` | The unit of planning. Confidence is linear (clamped, not Bayesian). `family` groups hypotheses by attack class so strategies can target the cluster, not the individual claim. |
| **Evidence** (`Evidence`) | `signal_type`, `polarity`, `verbatim_fragment`, `hypothesis_id?`, `confidence_delta` | Structured target signal extracted by the judge model. Polarity drives the support/refute edge automatically. |
| **Attempt** (`Attempt`) | `module`, `messages_sent`, `target_responses`, `experiment_id?`, `tested_hypothesis_ids[]`, `judge_score`, `outcome` | One module execution. Replaces `AttackNode` for belief-graph purposes — but with explicit links to the hypotheses tested, the strategy used, the evidence observed, and the experiment fulfilled. |
| **Strategy** (`Strategy`) | `family`, `template_summary`, `success_count`, `attempt_count` | Reusable attack pattern. Per-target stats here; global cross-target stats live in the [strategy library](./frontier-and-utility#the-strategy-library). |
| **Frontier experiment** (`FrontierExperiment`) | `hypothesis_id`, `module`, `instruction`, `expected_signal`, `state`, `utility` + 8 ranking components | A proposed next move. Ranked by [utility](./frontier-and-utility); the leader picks one by `id` and the manager that runs it records an `Attempt` that links back. |

Statuses live as small enums in [`mesmer/core/constants.py`](../../../../mesmer/core/constants.py):

- `HypothesisStatus`: ACTIVE → CONFIRMED (confidence ≥ **0.85**) /
  REFUTED (confidence ≤ **0.15**) / STALE (no fresh evidence in 3+
  runs).
- `Polarity`: SUPPORTS (raises confidence), REFUTES (lowers it),
  NEUTRAL (recorded but no shift).
- `EvidenceType`: ten signal categories — `refusal_template`,
  `partial_compliance`, `policy_reference`, `tool_reference`,
  `hidden_instruction_fragment`, `role_boundary_confusion`,
  `format_following_strength`, `objective_leak`,
  `refusal_after_escalation`, `unknown`.
- `ExperimentState`: PROPOSED → EXECUTING → FULFILLED / DROPPED.
- `AttemptOutcome`: LEAK, PARTIAL, REFUSAL, DEAD, OBJECTIVE_MET.

## Edges

Eleven typed edges, validated against an endpoint contract — apply a
delta whose edge violates the contract and `BeliefGraph.apply` raises
`InvalidDelta`.

| Edge kind | Source → Destination | Purpose |
|---|---|---|
| `hypothesis_supported_by_evidence` | hypothesis → evidence | Belief shifts up |
| `hypothesis_refuted_by_evidence` | hypothesis → evidence | Belief shifts down |
| `attempt_tests_hypothesis` | attempt → hypothesis | Audit: which beliefs this attempt probed |
| `attempt_used_strategy` | attempt → strategy | Audit: which pattern this attempt followed |
| `attempt_observed_evidence` | attempt → evidence | Audit: what the attempt produced |
| `attempt_confirmed_hypothesis` | attempt → hypothesis | Reserved for explicit confirm/refute attribution |
| `attempt_refuted_hypothesis` | attempt → hypothesis | Reserved for explicit confirm/refute attribution |
| `frontier_expands_hypothesis` | frontier → hypothesis | The experiment's intent |
| `frontier_uses_strategy` | frontier → strategy | The pattern the experiment will follow |
| `strategy_generalizes_from_attempt` | strategy → attempt | Lifts a successful attempt into a reusable pattern |
| `hypothesis_generalizes_to` | hypothesis → hypothesis | Reserved for hypothesis hierarchies |

Most edges are emitted automatically. An `EvidenceCreateDelta` whose
evidence has a non-NEUTRAL polarity emits the support/refute edge for
free; an `AttemptCreateDelta` emits one `attempt_tests_hypothesis` per
tested hypothesis, plus `attempt_used_strategy` and
`attempt_observed_evidence` as needed. Direct `EdgeCreateDelta` is
reserved for cross-cutting links the helper paths don't cover.

## The mutation contract

Every change to the belief graph is a typed `GraphDelta`. Twelve kinds
exist (one per `DeltaKind` value):

```
TARGET_TRAITS_UPDATE        STRATEGY_CREATE
HYPOTHESIS_CREATE           STRATEGY_UPDATE_STATS
HYPOTHESIS_UPDATE_CONFIDENCE FRONTIER_CREATE
HYPOTHESIS_UPDATE_STATUS    FRONTIER_UPDATE_STATE
EVIDENCE_CREATE             FRONTIER_RANK
ATTEMPT_CREATE              EDGE_CREATE
```

All mutations go through `BeliefGraph.apply(delta)`. No caller
mutates `graph.nodes` or `graph.edges` directly. Two consequences fall
out of this:

1. **The delta log doubles as a recovery log.** Replaying the deltas
   in order reconstructs the current state.
   `BeliefGraph.replay(delta_log_path)` rebuilds the graph from the
   JSONL when the snapshot is missing or corrupt.
2. **Deltas serialise from a snapshot, not from live state.** The
   apply paths deep-copy node payloads on insert. Mutating the live
   node (e.g. flipping a hypothesis status) doesn't bleed back into
   the audit log of the original create delta.

Bad deltas fail loud: `_require_node` checks every referenced id and
its kind; `_add_edge_validated` enforces the endpoint contract; status
flips on a non-existent hypothesis raise `InvalidDelta` rather than
silently dropping. The engine catches `InvalidDelta` at the boundary
and logs `BELIEF_DELTA` with the reason; deeper code lets the
exception propagate.

## Lifecycle of a run

**Bootstrap (run start, before the first dispatch).** When the active
hypothesis list is empty, [`runner.execute_run`](../../../../mesmer/core/runner.py)
calls `generate_hypotheses` (one judge-model LLM call, capped at four
hypotheses per call) and applies the resulting create deltas. Then
`generate_frontier_experiments` deterministically pairs each ACTIVE
hypothesis with the modules registered for its family — using two
lookup tables in [`agent/beliefs.py`](../../../../mesmer/core/agent/beliefs.py),
`_FAMILY_MODULE_HINTS` and `_FAMILY_EXPECTED_SIGNALS` — and emits
`StrategyCreateDelta` + `FrontierCreateDelta` per pairing. Finally
`rank_frontier` computes utility scores. **The frontier queue is
reducer-owned**: the LLM proposes hypotheses, but the planner queue
the leader has to dispatch is built without a second opaque model
call.

**Per target reply** (mid-attempt, after every successful target
reply in [`tools/send_message.py`](../../../../mesmer/core/agent/tools/send_message.py)).
A synthetic single-turn `Attempt` is built with `experiment_id` /
`tested_hypothesis_ids` resolved from `ctx.active_experiment_id`.
`extract_evidence` runs against that one exchange, the resulting
`Evidence` are applied, `apply_evidence_to_beliefs` shifts confidence
and flips status when thresholds are crossed, and `rank_frontier`
re-ranks the queue. The applied evidence list is summarised inline as
a `Belief evidence updated:` trailer in the tool result so the leader
sees mid-attempt belief shifts inside the ReAct loop, not just after
delegation returns. The turn index is recorded in
`ctx._belief_evidence_turn_indexes` so the module-level pass below
doesn't double-count it.

**Per attempt** (in [`evaluation._update_belief_graph`](../../../../mesmer/core/agent/evaluation.py),
after the sub-module concludes). An `AttemptCreateDelta` is applied
(auto-emitting all the audit edges). `extract_evidence` runs again
over the *unprocessed tail* of the transcript, evidence is applied,
beliefs are updated, the strategy gets a `StrategyUpdateStatsDelta`
with `success_inc=1` if the outcome is LEAK or OBJECTIVE_MET, and the
frontier is re-ranked. Each step's failure logs a typed
`BELIEF_DELTA` / `EVIDENCE_EXTRACT_ERROR` event but does not abort
the run — a single failed extractor call can't strand the planner.

**Run end.** Per-target `Strategy` nodes (those with at least one
attempt) are folded into the [global cross-target library](./frontier-and-utility#the-strategy-library)
at `~/.mesmer/global/strategies.json`. Counters add; trait
correlations dedupe-merge.

## Persistence

The belief graph lives next to the legacy attack graph, distinct
files:

```
~/.mesmer/targets/{target-hash}/
├── belief_graph.json    # current snapshot (authoritative for read)
├── belief_deltas.jsonl  # append-only audit log (replay-able)
└── graph.json           # legacy AttackGraph (untouched)
```

The snapshot is the authoritative current state.
`BeliefGraph.save(snapshot_path, delta_log_path=...)` writes the
snapshot **and** appends every queued delta as one JSON line. The
JSONL log is the audit trail; if it diverges from the snapshot, the
snapshot wins on the next load. `BeliefGraph.replay(delta_log_path)`
reconstructs the graph from the log alone — the recovery path when
the snapshot is missing or corrupt.

Target hashing is the same as the legacy graph
(`sha256(adapter|url|model)` → 16-char hex), so the same
`{target-hash}` directory holds both. `--fresh` bypasses loading both
files; the disk copies are left untouched and the next run without
`--fresh` sees the original state again.

## Role-scoped briefs

The belief graph is injected into module prompts by
[`GraphContextCompiler`](../../../../mesmer/core/agent/graph_compiler.py).
One brief per `BeliefRole`:

| Role | What the brief contains | Audience |
|---|---|---|
| `LEADER` | Full belief landscape (top hypotheses with confidence + strongest evidence each), ranked frontier (★ marks the [selector's pick](./frontier-and-utility#ucb-with-lookahead)), dead zones, required-action contract naming `experiment_id` | The synthesised executive |
| `MANAGER` | Only the active experiment + the hypothesis it tests + supporting/refuting evidence + report-back contract | Depth-1 manager modules |
| `EMPLOYEE` | Focused job description: probe, observe, conclude with transcript; do not decide objective success | Depth-≥2 employee modules |
| `JUDGE` | Hypothesis slate so the in-loop judge can score against belief shifts, plus expected_signal for the active experiment | The judge LLM at scoring time |
| `EXTRACTOR` | Hypothesis slate so the extractor tags evidence with the right id (mostly symmetry — the extractor builds its own slate inline) | The evidence extractor |

`token_budget` is a soft cap; the renderer trims trailing sections
(oldest evidence, lowest-utility experiments) until the brief fits.
The leader brief flagged with `★` looks like:

```
## Recommended Experiments
★ fx_a1b2c3d4 · format-shift · u=0.71 · expand wh_e5f6g7h8 (0.42)
  fx_d9e8f7a6 · authority-bias · u=0.58 · expand wh_b3c4d5e6 (0.31)
  fx_…
```

Selector picks are advisory; the leader can override.

## Side-by-side with the legacy attack graph

| | Attack graph | Belief graph | [Strategy library](./frontier-and-utility#the-strategy-library) |
|---|---|---|---|
| **File** | `~/.mesmer/targets/{hash}/graph.json` | `~/.mesmer/targets/{hash}/belief_graph.json` + `belief_deltas.jsonl` | `~/.mesmer/global/strategies.json` |
| **Tracks** | Per-target attempt history | Per-target weakness beliefs + evidence + frontier | Cross-target attack patterns |
| **Used for** | Legacy frontier proposer, dead-end pruning, tier gating, audit trail, the [Attack Graph view](../recipes/the-web-editor) | Hypothesis confidence, utility ranking, role-scoped briefs, the [Belief Map view](../recipes/reading-the-belief-map) | Bootstrap context for hypothesis generation, `transfer_value` component of utility |

Both graphs are persisted; both are read by the engine on every run.
The web UI's toggle pill at the top of the graph area flips between
the two views — same data folder, different lens.
