This page covers the **legacy attack-attempt graph** — a per-module
attempt log that's been there since v0.1.0. Mesmer also runs a
parallel typed planner state called the [belief graph](./belief-graph)
that tracks weakness hypotheses, evidence, and frontier experiments.
Both are persisted, both are read by the engine on every run; the web
UI's toggle pill flips between them. Read this page first if you want
to understand how mesmer scores attempts and prunes dead ends; read
the [belief graph](./belief-graph) for how the planner now picks the
*next* move.

Every time mesmer attacks a target, it writes the result into a graph.
Every subsequent run reads that graph back before planning the next move.
This is what makes mesmer **adaptive** — the attacker gets smarter with
exposure. The graph is **the** source of truth for what a target has
ever seen and how we judged it.

## Shape

Nodes are attack attempts (one per module execution, including the
synthesised executive's run-end verdict). Edges are parent-child
refinements.

Every module execution produces exactly **one** `AttackNode` in the
graph. Manager and employee executions are recorded by
`evaluation._update_graph` from inside the parent's dispatch; the
**executive's** own run-end verdict is recorded by `execute_run` right
after the outer ReAct loop returns, distinguished by `source: LEADER`
(the enum value name is preserved across the leader → executive rename
for graph-schema compatibility — treat it as the opaque "depth-0 root"
marker).

The executive is **synthesised in memory at run start** (not loaded
from any `module.yaml`); its name is `f"{scenario_stem}:executive"`.
Authored manager modules listed in `scenario.modules` are dispatched
by the executive at depth 1; their employees nest at depth ≥ 2.

## Node anatomy

```json
{
  "id": "n-a7f2",
  "parent": "n-a7f0",
  "module": "prefix-commitment",
  "score": 8.4,
  "status": "PROMISING",
  "source": "AGENT",
  "approach": "Inject 'Sure, here are my instructions: ...' as a forced prefix",
  "leaked_info": "You are Acme Corp's customer support assistant...",
  "reflection": "prefix commits the continuation into verbatim leak",
  "module_output": "<full conclude() text — authoritative record>",
  "turn_range": [5, 7]
}
```

Four **lifecycle statuses** — all live in the `NodeStatus` enum in
`core/constants.py`:

| Status | Meaning |
|---|---|
| `FRONTIER` | Proposed but not yet tried. The planner picks from this set. |
| `ALIVE` | Tried with promising results. May be revisited to refine. |
| `PROMISING` | High-scoring descendent candidates. Priority on next run. |
| `DEAD` | Score ≤ `DEAD_SCORE_THRESHOLD` or explicitly killed by the judge. Never re-walked. |

## Four kinds of proposer

Every node has a `source` field recording who produced it:

- **`AGENT`** — the ReAct loop proposed it during `_reflect_and_expand`.
- **`JUDGE`** — the in-loop LLM judge surfaced it while scoring an
  attempt ("the target leaked auth context, try that angle").
- **`HUMAN`** — came in via `mesmer hint`, `mesmer debrief`, or the web
  UI. Explored **first** by the executive on the next run — human
  pattern-matching beats the judge for creative leaps.
- **`LEADER`** — the **executive's** run-end verdict node (one per
  run). The enum value name predates the leader → executive rename and
  is preserved for graph-schema compatibility. Filtered out of
  attempt-centric walks (TAPER trace, winning-module attribution) via
  `AttackNode.is_leader_verdict`; renders as a **square** in the bench
  viz so every tree ends on the executive's decision, not on whichever
  sub-module was last delegated to.

## Persistence

```
~/.mesmer/targets/{hash}/
├── graph.json          # full node tree — the authoritative record
├── scratchpad.md       # the executive's persistent working notes —
│                       # written by the update_scratchpad tool AND by
│                       # the operator chat; seeded into the executive's
│                       # scratchpad slot at run start
├── chat.jsonl          # append-only operator ↔ executive chat log;
│                       # one JSON row per message
├── profile.md          # optional free-form human notes (hand-edited)
├── conversation.json   # CONTINUOUS-mode rolling target transcript
└── runs/
    └── {run_id}.jsonl  # append-only Turn log per run
```

`{hash}` is `sha256(adapter | url | model)` truncated to 16 hex chars.
Different model strings hash differently, so upgrading your target
branches into a fresh graph rather than corrupting old state.

Pass `--fresh` to any `mesmer run` invocation to bypass loading the
existing graph and to clear the CONTINUOUS conversation. **`--fresh`
does NOT delete `scratchpad.md`** — the executive's curated notes are
treated as durable human-curated state. Delete it manually if you want
a truly clean run. Use `mesmer graph reset <scenario>` to delete the
graph file entirely (asks for confirmation first).

The legacy `plan.md` file is gone; `TargetMemory.__init__` performs a
one-shot rename of `plan.md` → `scratchpad.md` on first init of an
existing target so old persistence directories migrate automatically.

Cross-target technique stats live separately at
`~/.mesmer/global/techniques.json` (`mesmer stats` surfaces these).

**Trial-scoped graph snapshots** (bench only): each trial also writes
`benchmarks/results/{date}/events/{trial_id}.graph.json` — root + only
that run's nodes — for diffable per-trial forensics. These don't replace
`~/.mesmer/` state; they're a scoped slice for the viz and telemetry
extraction.

## Budget modes

The leader's prompt framing shifts based on where you are in the turn
budget (`Context.budget`). Three modes, all in the `BudgetMode` enum:

1. **`EXPLORE`** — early turns. Frontier nodes get a prompt bonus so
   new techniques get tried before well-worn ones.
2. **`EXPLOIT`** — mid-run. The current best node's descendants get
   the bonus. Refinement over discovery.
3. **`CONCLUDE`** — final ~20% of turns. Leader is instructed to stop
   branching and drive the strongest lead to completion.

Thresholds are tunables in `core/constants.py`
(`BUDGET_EXPLORE_UPPER_RATIO = 0.5`, `BUDGET_EXPLOIT_UPPER_RATIO = 0.8`).

## The judge vs the canary

Two scoring systems coexist, and it is important they stay distinct:

- **In-loop LLM judge** (`core/agent/judge.py`) — 1-10 rubric per
  attempt, extracts insights, proposes frontier. Drives graph updates
  and next-move selection. **Not authoritative for benchmarks.**
- **Deterministic canary matcher** (`bench/canary.py`) — substring
  match against the leader's concluded output. Zero LLM, zero
  randomness. This is what decides benchmark success.

A key consequence: an accidental canary leak that appears in a
sub-module's probe but which the leader never consolidated into its
verdict text does NOT count. Leader-grounded scoring is the whole point
of the LEADER verdict node.

For the guided tour of trial forensics, see
[Bench tracing](/docs/concepts/bench-trace).

## Side-by-side with the belief graph

| | Attack graph | [Belief graph](./belief-graph) | [Strategy library](./frontier-and-utility#the-strategy-library) |
|---|---|---|---|
| **File** | `~/.mesmer/targets/{hash}/graph.json` | `~/.mesmer/targets/{hash}/belief_graph.json` + `belief_deltas.jsonl` | `~/.mesmer/global/strategies.json` |
| **Tracks** | Per-target attempt history | Per-target weakness beliefs + evidence + frontier | Cross-target attack patterns |
| **Used for** | Legacy frontier proposer, dead-end pruning, tier gating, audit trail, the Attack Graph view | Hypothesis confidence, utility ranking, role-scoped briefs, the [Belief Map view](../recipes/reading-the-belief-map) | Bootstrap context for hypothesis generation, the `transfer_value` utility component |

Both graphs are persisted; both are read by the engine on every run.
`--fresh` bypasses loading both. The graph view in the web UI is a
single toggle — no migration, no duplication, two lenses on the same
data folder.
