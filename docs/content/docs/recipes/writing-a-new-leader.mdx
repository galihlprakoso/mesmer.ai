---
title: Writing a new leader
description: When to author a new leader vs reuse system-prompt-extraction. How to pick sub_modules and when to set see_siblings.
---

A leader is a module that **orchestrates** sub-modules to satisfy an
objective. `system-prompt-extraction` is the only leader that ships
today; it covers most "extract a hidden string from a target"
objectives. Author a new leader when the objective shape itself is
different — not when you just want different techniques.

## When to NOT write a new leader

Most "I want mesmer to do X" turns out to be a sub-module change, not
a leader change. Skip the new leader when:

- You want different techniques in the rotation. Just edit
  `system-prompt-extraction.yaml`'s `sub_modules:` list, or write the
  technique and add it.
- You want a different judge rubric. Use `judge_rubric_additions:` on
  the scenario; works without touching modules.
- You want a different success criterion. Edit the scenario's
  `objective.goal` text — every module sees it.

## When you DO need one

Real "new leader" cases:

- **Different orchestration shape.** Your objective needs a different
  three-phase flow than recon → plan → execute. E.g. a
  policy-violation extractor might want trial → escalate → consolidate.
- **Different judge rubric across phases.** The leader's own
  `judge_rubric:` shapes how the judge scores attempts in *this
  leader's* runs. If your objective scores differently from extraction,
  fork the leader.
- **Different sub-module roster.** You're attacking a fundamentally
  different surface (e.g. tool-call abuse instead of system-prompt
  extraction) and almost none of the existing techniques apply.

## The skeleton

```yaml title="modules/attacks/your-leader/module.yaml"
name: your-leader
tier: 0                        # leaders are always T0 — they orchestrate, don't probe.

description: |
  One-line summary of what objective this leader satisfies. Visible
  in `mesmer modules list` and the web editor's leader picker.

theory: |
  Why this objective needs its own leader (vs. reusing
  system-prompt-extraction). 1-2 sentences.

system_prompt: |
  You are the leader for <objective shape>. Your job is orchestration,
  not attack craft. Read the scratchpad. Pick the next sub-module to
  delegate. Read the judge verdict. Decide whether to continue,
  replan, or conclude.

  ## Phase 0 — RECON
  1. <when to run target-profiler>
  2. <what dossier you expect>

  ## Phase 1 — PLAN
  1. <when to run attack-planner>
  2. <how to read the plan>

  ## Phase 2 — EXECUTE
  1. <how you pick the next step>
  2. <when you replan vs continue>
  3. <when you conclude(objective_met=true)>

sub_modules:
  - target-profiler              # T0 recon (almost always)
  - name: attack-planner
    see_siblings: true           # the planner needs the sibling roster
  # ─── techniques you actually want this leader to use ───
  - direct-ask                   # T0
  - prefix-commitment            # T1
  - your-new-technique           # T2

judge_rubric: |
  This leader's domain-specific scoring. Tell the judge what counts
  as a "good attempt" for THIS leader's objective shape.

reset_target: false
```

## `see_siblings` — when to set it

The sibling roster (every other module in this leader's `sub_modules:`,
with names + descriptions + theories) is **not** included by default
in any sub-module's prompt. To inject it, set `see_siblings: true` on
the sub-module entry:

```yaml
sub_modules:
  - name: attack-planner
    see_siblings: true        # planner can name siblings in its plan
  - target-profiler           # bare string — no siblings injected
```

Use `see_siblings: true` for any sub-module that *reasons about which
siblings to recommend*. The canonical case is `attack-planner`. Leave
it false (or omit) for techniques that just probe — they don't need
to know about siblings.

`call_siblings: true` is parsed but not yet wired to expose siblings
as callable tools. Reserved.

## Mode and reset semantics

- Leaders are T0 — they don't run techniques themselves.
- Leaders should leave `reset_target: false`. The leader owns the
  whole run; resetting between leader iterations would defeat
  CONTINUOUS mode.
- Sub-modules under your leader inherit the leader's mode. CONTINUOUS
  mode forces `reset_target: false` for all sub-modules regardless
  of their declaration (logged as `MODE_OVERRIDE`).

## Test the orchestration

```bash
# Verify the leader loads and the sub-module list resolves
uv run mesmer modules describe your-leader

# Use it in a scenario
cat > scenarios/private/test-your-leader.yaml <<EOF
name: "Test your-leader"
mode: trials

target:
  adapter: openai
  base_url: https://openrouter.ai/api/v1
  model: openai/gpt-4o-mini
  api_key: "${OPENROUTER_API_KEY}"

objective:
  goal: <your objective text>
  max_turns: 15

module: your-leader

agent:
  model: openrouter/anthropic/claude-sonnet-4
  api_key: "${OPENROUTER_API_KEY}"
EOF

uv run mesmer run scenarios/private/test-your-leader.yaml --verbose
```

Watch for `delegate` events — every line shows which sub-module the
leader picked and its tier. If the leader keeps freestyling
techniques the planner didn't recommend, `attack-planner` probably
isn't getting the sibling roster (`see_siblings: true` missing).

## Don't write the OBJECTIVE AWARENESS stanza

Same rule as techniques: the engine appends the depth-aware stanza
automatically. The leader gets the `conclude(result=...,
objective_met=true)` protocol. Don't write `OBJECTIVE MET —` strings
into the leader's `system_prompt`.
