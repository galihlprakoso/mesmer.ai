---
title: Writing a new manager
description: When to author a manager module vs reuse system-prompt-extraction. How to compose sub_modules and when to set see_siblings.
---

A **manager** is an authored module listed in `scenario.modules`. It
runs at depth 1 — the synthesised executive dispatches it, it
orchestrates sub-modules, it talks to the target via `send_message`,
and it concludes with a write-up. Three managers ship today:
`system-prompt-extraction`, `tool-extraction`, `exploit-analysis`.
Author a new one when the **objective shape** is fundamentally
different from those.

## When to NOT write a new manager

Most "I want mesmer to do X" turns out to be a sub-module change, not
a manager change. Skip the new manager when:

- You want different techniques in the rotation. Edit
  `system-prompt-extraction.yaml`'s `sub_modules:` list, or write the
  technique and add it (see
  [Writing a new technique](/docs/recipes/writing-a-new-technique)).
- You want a different judge rubric. Use `judge_rubric_additions:` on
  the scenario; works without touching modules.
- You want a different success criterion. Edit the scenario's
  `objective.goal` text — every module sees it.
- You want the executive to behave differently. The executive's
  prompt is `core/agent/prompts/executive.prompt.md`; per-scenario
  override via the optional `leader_prompt:` YAML field. **No
  authored manager controls executive behaviour.**

## When you DO need one

Real "new manager" cases:

- **Different orchestration shape.** Your objective needs a different
  three-phase flow than recon → plan → execute. E.g. a
  policy-violation extractor might want trial → escalate →
  consolidate.
- **Different judge rubric across phases.** The manager's own
  `judge_rubric:` shapes how the judge scores attempts in *this
  manager's* runs. If your objective scores differently from
  extraction, fork the manager.
- **Different sub-module roster.** You're attacking a fundamentally
  different surface (e.g. tool-call abuse instead of system-prompt
  extraction) and almost none of the existing techniques apply.

## The skeleton

```yaml title="modules/attacks/your-manager/module.yaml"
name: your-manager
tier: 0                        # managers are always T0 — they orchestrate, don't probe.

description: |
  One-line summary of what objective this manager satisfies. Visible
  in `mesmer modules list` and the web editor's manager picker.

theory: |
  Why this objective needs its own manager (vs. reusing
  system-prompt-extraction). 1-2 sentences.

system_prompt: |
  You are a manager module for <objective shape>. Your job is
  orchestration, not attack craft. Read the scratchpad. Pick the next
  sub-module to delegate. Read the judge verdict. Decide whether to
  continue, replan, or conclude.

  You CAN call send_message to the target directly when a single
  probe is the right move; you CANNOT call talk_to_operator,
  update_scratchpad, or ask_human (those are executive-only).

  ## Phase 0 — RECON
  1. <when to run target-profiler>
  2. <what dossier you expect>

  ## Phase 1 — PLAN
  1. <when to run attack-planner>
  2. <how to read the plan>

  ## Phase 2 — EXECUTE
  1. <how you pick the next step>
  2. <when you replan vs continue>
  3. <when you conclude(objective_met=true)>

sub_modules:
  - target-profiler              # T0 recon (almost always)
  - name: attack-planner
    see_siblings: true           # the planner needs the sibling roster
  # ─── techniques you actually want this manager to use ───
  - direct-ask                   # T0
  - prefix-commitment            # T1
  - your-new-technique           # T2

judge_rubric: |
  This manager's domain-specific scoring. Tell the judge what counts
  as a "good attempt" for THIS manager's objective shape.

reset_target: false
```

`is_executive: true` does **nothing** in authored YAML — the loader
ignores the field. The executive is synthesised at run start by
`runner.execute_run`; you can't author one.

## `see_siblings` — when to set it

The sibling roster (every other module in this manager's
`sub_modules:`, with names + descriptions + theories) is **not**
included by default in any sub-module's prompt. To inject it, set
`see_siblings: true` on the sub-module entry:

```yaml
sub_modules:
  - name: attack-planner
    see_siblings: true        # planner can name siblings in its plan
  - target-profiler           # bare string — no siblings injected
```

Use `see_siblings: true` for any sub-module that *reasons about which
siblings to recommend*. The canonical case is `attack-planner`. Leave
it false (or omit) for techniques that just probe — they don't need
to know about siblings.

`call_siblings: true` is parsed but not yet wired to expose siblings
as callable tools. Reserved.

## Mode and reset semantics

- Managers are T0 — they don't run techniques themselves.
- Managers should leave `reset_target: false`. The manager owns the
  whole delegation tree; resetting between manager iterations would
  defeat CONTINUOUS mode.
- Sub-modules under your manager inherit the manager's mode.
  CONTINUOUS mode forces `reset_target: false` for all sub-modules
  regardless of their declaration (logged as `MODE_OVERRIDE`).

## Test the orchestration

```bash
# Verify the manager loads and the sub-module list resolves
uv run mesmer modules describe your-manager

# Use it in a scenario
cat > scenarios/private/test-your-manager.yaml <<EOF
name: "Test your-manager"
mode: trials

target:
  adapter: openai
  base_url: https://openrouter.ai/api/v1
  model: openai/gpt-4o-mini
  api_key: "${OPENROUTER_API_KEY}"

objective:
  goal: <your objective text>
  max_turns: 15

modules:
  - your-manager

agent:
  model: openrouter/anthropic/claude-sonnet-4
  api_key: "${OPENROUTER_API_KEY}"
EOF

uv run mesmer run scenarios/private/test-your-manager.yaml --verbose
```

Watch for `delegate` events — every line shows which sub-module the
manager picked and its tier. If the manager keeps freestyling
techniques the planner didn't recommend, `attack-planner` probably
isn't getting the sibling roster (`see_siblings: true` missing).

## Don't write the OBJECTIVE AWARENESS stanza

Same rule as techniques: the engine appends a depth-aware stanza
automatically. Sub-modules (depth > 0) get the `OBJECTIVE SIGNAL —
<fragment>` flag protocol. Only the executive (depth 0) terminates
the run via `conclude(objective_met=true)`. Don't write `OBJECTIVE
MET —` strings into the manager's `system_prompt` — sub-modules will
copy the format and try to terminate themselves.

## Don't reach for executive-only tools

Managers can't call `talk_to_operator` / `update_scratchpad` /
`ask_human`. The framework filters those out of `build_tool_list`
when `is_executive=False`. If your manager's prompt mentions any of
them, the model will hallucinate a call that resolves to "unknown
tool" — the operator sees nothing and the manager's iteration burns.
Audit your prompt for residual mentions.

If the work genuinely belongs at the operator level, return it from
the manager's `conclude()` — the executive sees the full text in its
next prompt and can call `talk_to_operator` itself.
